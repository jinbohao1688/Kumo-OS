# Kumo OS 项目规划文档

> 目标：在 Ubuntu 22.04 上用 C 语言从零开发一个"可启动、可交互、可扩展"的独立操作系统。
> 原则：先规划、后实现；先跑起来、再迭代；诚实评估风险，不画饼。

---

## 0. 总路线图（一图流，文字版）

```
阶段0 环境搭建        → 产出：能编译一个跑在 QEMU 里的 hello.bin
阶段1 引导+屏幕输出    → 产出：GRUB 拉起内核，内核打印字符串
阶段2 GDT/IDT/中断异常 → 产出：能捕获 除零/缺页 等异常并打印现场
阶段3 内存管理(分页)   → 产出：物理页分配器 + 内核态虚拟地址空间
阶段4 多任务调度       → 产出：多个内核线程轮转执行，时钟中断驱动调度
阶段5 用户态+syscall   → 产出：从 Ring0 切到 Ring3，int 0x80 系统调用
阶段6 文件系统         → 产出：内存文件系统或简单块设备 FS，可读写文件
阶段7 shell+用户程序   → 产出：能敲命令、加载并运行简单用户态程序
阶段8 驱动扩展         → 产出：键盘/磁盘/串口驱动完善，可选网络/图形

主线依赖关系（不可跳步）：
0 → 1 → 2 → 3 → 4 → 5 → (6 与 7 部分并行) → 8
```

**现实预期**：这是一个"个人/小团队长期项目"的节奏，不是几周能完成的东西。业界经验值：一个人利用业余时间，从 0 到"能跑 shell、有基础文件系统"大约需要 **6～12 个月**持续投入（假设每周 10～15 小时），中间还会有大量调试黑屏、三重 fault、无限重启的时间。这不是吓退你，而是让你把节奏定对——**目标是可持续推进，不是憋一个大版本**。

## 1. 项目总览

### 1.1 分层架构（文字架构图）

```
┌─────────────────────────────────────────┐
│  用户态：shell、用户程序 (ELF 可执行文件)   │  Ring 3
├─────────────────────────────────────────┤
│  系统调用接口 (syscall/int 0x80)          │  ← 用户态/内核态边界
├─────────────────────────────────────────┤
│  内核子系统：                              │  Ring 0
│  ├─ 进程/线程调度器 (sched)                │
│  ├─ 虚拟文件系统 VFS + 具体 FS (fs)        │
│  ├─ 内存管理：物理页帧 + 分页 + 堆 (mm)     │
│  ├─ 设备驱动：键盘/串口/磁盘/屏幕 (drivers) │
│  └─ 中断/异常处理：IDT、PIC/APIC (arch)     │
├─────────────────────────────────────────┤
│  硬件抽象层：GDT/IDT/分页表操作等 (arch)    │
├─────────────────────────────────────────┤
│  引导层：GRUB (Multiboot2) → 内核入口       │  boot/
├─────────────────────────────────────────┤
│  硬件 / QEMU 虚拟机                        │
└─────────────────────────────────────────┘
```

层与层之间的关系是**严格自底向上依赖**的：调度器依赖内存管理（每个任务要有独立栈/页表），syscall 依赖调度器（要知道是哪个进程在调用），文件系统依赖内存管理（缓存）和驱动（磁盘 I/O），shell 依赖 syscall 和文件系统。这也是后面阶段顺序的根本原因。

### 1.2 哪些必须自己写，哪些可以借现成工具

**必须自己写（这是"操作系统"这三个字的核心）：**
- 内核本体：中断处理、内存管理、调度器、syscall 分发
- 文件系统的内核侧实现（哪怕极简）
- 设备驱动（键盘、屏幕/串口、磁盘）
- shell（命令解析、内建命令、进程加载）

**可以/应该借助现成工具（重新发明这些没有教学价值，纯粹浪费时间）：**
- **引导器**：用 GRUB2，不要自己写 MBR 引导代码。自己写 bootloader 只在你想专门学"引导原理"时才有必要，否则会拖慢整个项目且价值有限。
- **交叉编译工具链**：用 GCC 交叉编译器（x86_64-elf-gcc），不要试图用系统自带 gcc 硬撸（会被 host 相关的库/ABI 坑死）。
- **调试运行环境**：QEMU + GDB，不要一上来烧真机。
- **构建系统**：Makefile（或后期 CMake），不必自研。
- **libc**：早期完全不需要 libc；到用户态阶段可以移植极简 libc（如 newlib 的裁剪版）而不是自己写完整 libc。

### 1.3 最合理的技术路线

结论先行：**x86_64、GRUB2 Multiboot2 引导、C + 少量内联/独立汇编、QEMU 优先、裸机(freestanding) + 自研极简运行时**。

理由：
1. x86_64 资料最全（OSDev Wiki、Intel/AMD 手册、各类教程绝大多数以 x86 为例），遇到问题最容易搜到答案，这对个人项目的可持续性至关重要。
2. GRUB2 是事实标准引导器，Multiboot2 协议简单可靠，能让你把精力放在内核而不是引导协议细节上。先用 32 位保护模式起步再跳 64 位长模式也是经典路径（见下方选型细节）。
3. QEMU 提供快速的"改代码→重启验证"循环（秒级），远快于真机烧录，这是保证项目能持续推进的关键基础设施。

## 2. 技术选型建议

| 选型项 | 建议 | 理由 |
|---|---|---|
| 宿主编译器 | GCC（Ubuntu 自带） | 用于写工具链本身、写宿主测试代码 |
| 目标交叉编译器 | `x86_64-elf-gcc`（自己用 crosstool-NG 或脚本编译） | 系统自带 gcc 默认链接宿主 libc/glibc ABI，直接编译内核会引入你不想要的依赖和 ABI 假设；交叉编译器保证生成纯粹的 freestanding 代码 |
| 汇编器 | NASM | 语法比 GAS 的 AT&T 语法更直观，OSDev 教程主流用 NASM，社区资料多 |
| 链接器 | GNU ld（交叉工具链自带），配合自定义 linker script | 需要精确控制内核加载地址、段布局 |
| 引导方式 | GRUB2 + Multiboot2 规范 | 不自己写 bootloader；GRUB 帮你完成实模式→保护模式切换、加载 ELF 到内存等脏活 |
| 架构位宽 | **先 32 位保护模式起步 → 阶段2/3 之间过渡到 64 位长模式** | 32 位保护模式的 GDT/分页/中断模型更简单，资料（如经典的 "Roll your own toy UNIX"、JamesM's kernel tutorial）都以 32 位讲解；吃透后再切 64 位（多一层 PML4、寄存器行为略有不同），学习曲线更平滑。如果你已经很熟 x86，也可以直接上 64 位，代价是初期调试信息更复杂 |
| 是否裸机 | 是（freestanding，`-ffreestanding -nostdlib`） | OS 内核不能依赖宿主 libc（没有 malloc/printf 这些运行时支撑），必须自己实现最小运行时 |
| 标准库替代 | 自己实现极简 `kstring.c`（memcpy/memset/strlen等）+ 后期考虑移植裁剪版 newlib 给用户态用 | 内核态坚决不用标准库；用户态在阶段6/7可以引入极简 libc 减轻应用开发负担 |
| 模拟器 | **必须先做 QEMU 环境，且贯穿全程** | 真机调试成本极高（烧盘、重启、无 GDB），QEMU 支持 `-s -S` 直接接 GDB 单步调试内核，这是新手能不能坚持下去的关键 |

**关于 32 位 vs 64 位的补充说明（这是容易踩坑的决策点）**：不少教程会让你纠结"要不要一步到位写64位"。经验建议：如果你是第一次写OS，先在32位保护模式把 阶段1~3（引导、中断、分页）跑通，理解了这套机制后，切64位只是"多一层页表 + 寄存器变宽 + GDT略微不同"，心智负担小很多。如果你直接上64位，长模式的进入过程本身（先进32位保护模式→再开PAE→再跳长模式）就是一段不小的复杂度，容易在阶段1就卡住打击积极性。

## 3. 分阶段开发计划

### 阶段 0：环境搭建
- **目标**：搭好一套"改代码→编译→QEMU运行→GDB调试"的完整闭环。
- **模块**：无内核代码，纯工具链。
- **完成标准**：能在 QEMU 里跑起一个用 NASM 写的、只做 `写显存打印字符` 的裸机 `.bin`，且能用 GDB attach 上去单步。
- **可能踩坑**：
  - Ubuntu 22.04 自带 gcc 版本与交叉编译工具链构建脚本不兼容（需要匹配 binutils/gcc 版本组合）。
  - QEMU 版本差异导致 `-kernel` 参数行为不同。
  - GDB 架构不匹配（需要 `gdb-multiarch` 而不是普通 gdb）。
- **学习资料方向**：OSDev Wiki「GCC Cross-Compiler」页面；不需要看书，跟着 wiki 操作一遍即可。

### 阶段 1：成功启动并打印内容
- **目标**：GRUB 引导你的内核 ELF，内核在屏幕上打印字符串（VGA文本模式或串口）。
- **模块**：`boot/`（Multiboot2 头）、`arch/x86/boot.asm`（入口跳转到 C）、`kernel/main.c`、`drivers/vga.c` 或 `drivers/serial.c`。
- **完成标准**：`qemu-system-i386 -cdrom kumo.iso` 后屏幕显示 "Kumo OS booted."。
- **可能踩坑**：
  - Multiboot2 头对齐/校验和算错，GRUB 直接拒绝加载（黑屏无提示，最容易劝退的一步）。
  - 忘记设置好栈指针就跳进 C 代码，导致函数调用直接崩。
  -链接脚本（linker script）里入口地址、段地址算错，GRUB 加载了但跳到了错误地址。
- **建议**：**先用串口输出而不是VGA文本模式做主要调试通道**——串口在QEMU里可以直接重定向到终端/文件，比截图看VGA屏幕更适合长期调试和自动化测试。
- **学习资料方向**：OSDev Wiki「Bare Bones」教程（这是几乎所有人的起点）；理解 ELF 格式基础。

### 阶段 2：中断、异常、内核基础框架
- **目标**：建立 GDT、IDT，能捕获硬件异常（除零、缺页、GPF等）并打印现场信息而不是直接死机重启。
- **模块**：`arch/x86/gdt.c`、`arch/x86/idt.c`、`arch/x86/isr.asm`（中断服务例程入口）、`arch/x86/pic.c`（8259 PIC 重映射）。
- **完成标准**：故意触发一次除零异常，内核打印出寄存器现场（EIP/EAX等）而不是三重 fault 重启；时钟中断（IRQ0）能被捕获并周期性打印 tick。
- **可能踩坑**：
  - PIC 不重映射会导致硬件中断和 CPU 异常号冲突（经典坑，教程里必讲）。
  - IDT 表项的 selector/权限位设置错误，进中断服务例程直接 GPF。
  - 中断服务例程用汇编写 trampoline 时忘记保存/恢复全部寄存器现场。
- **学习资料方向**：Intel SDM Vol.3 第6章（中断与异常）；OSDev Wiki「Interrupts」「IDT」「Exceptions」。

### 阶段 3：内存管理
- **目标**：从"能用的物理内存有多大都不知道"到"有物理页帧分配器 + 分页开启 + 内核堆"。
- **模块**：`mm/pmm.c`（物理内存管理，位图法起步）、`mm/paging.c`（页表建立、`CR3`加载）、`mm/kheap.c`（内核堆，简单 bump allocator 或链表分配器）。
- **完成标准**：能动态分配/释放物理页；开启分页后内核仍正常运行（说明恒等映射或高半内核映射正确）；`kmalloc`/`kfree` 可用且不崩溃。
- **可能踩坑**：
  - 开启分页瞬间三重 fault（映射没做对，PC直接飞了）——这是新手死亡率最高的一步之一。
  - Multiboot 提供的内存map解析错误，把内核自己占用的内存标记成"空闲"导致后续分配覆盖内核代码。
  - 高半内核（higher-half kernel）映射的链接脚本/页表配合出错。
- **学习资料方向**：OSDev Wiki「Memory Management」「Paging」「Higher Half Kernel」；这一阶段建议不要跳过"先恒等映射跑通，再改高半内核"这个中间步骤。

### 阶段 4：多任务/调度
- **目标**：多个内核态"任务"能被时钟中断打断并轮转执行（先不涉及用户态）。
- **模块**：`sched/task.c`（任务控制块 TCB）、`sched/context_switch.asm`（寄存器/栈切换）、`sched/scheduler.c`（简单轮转调度）。
- **完成标准**：创建2~3个内核线程，每个死循环打印自己的编号，时钟中断触发下能看到编号交替打印。
- **可能踩坑**：
  - 上下文切换汇编里栈操作顺序错，切回来的任务栈是坏的。
  - 调度发生在中断上下文里，忘记正确处理"中断嵌套"或"调度时中断应保持关闭"。
  - 每个任务的独立内核栈忘记单独分配，共用一个栈导致互相踩踏。
- **学习资料方向**：OSDev Wiki「Kernel Multitasking」；可以参考 xv6（MIT教学OS）的调度器实现思路（不建议直接抄代码，但架构思路值得学）。

### 阶段 5：文件系统与系统调用
- **目标**：建立用户态/内核态边界（Ring3切换）+ syscall 分发机制 + 一个极简文件系统。
- **模块**：`arch/x86/usermode.asm`（Ring0→Ring3切换，TSS配置）、`kernel/syscall.c`（syscall分发表）、`fs/vfs.c`（统一接口）、`fs/ramfs.c`（先做内存文件系统，比磁盘FS简单得多）。
- **完成标准**：一个用户态"程序"（先内嵌在内核镜像里，不走磁盘加载）能执行 `write` syscall 把字符串打印出来；ramfs 能创建/读/写文件。
- **可能踩坑**：
  - TSS（任务状态段）配置错误导致 Ring3→Ring0 时栈切换失败（这是从内核态到用户态最容易卡住的一步）。
  - syscall 参数传递约定（寄存器 vs 栈）在用户态和内核态两侧不一致。
  - 忘记做权限检查，用户态代码能直接访问内核内存（先能跑通功能，权限隔离作为后续加固项，但要明确记录这是已知欠账）。
- **学习资料方向**：OSDev Wiki「Getting to Ring 3」「System Calls」；先做 ramfs 再考虑 FAT32 之类真实磁盘文件系统（磁盘FS涉及块设备驱动，复杂度陡增，不要在这一阶段同时啃）。

### 阶段 6：shell 与基础用户态程序
- **目标**：一个能敲命令、执行内建命令、（有余力的话）加载执行简单ELF程序的 shell。
- **模块**：`shell/shell.c`（命令行读取/解析）、`shell/builtin.c`（内建命令：ls/cat/echo/help等）、`user/init.c`（第一个用户态进程）。
- **完成标准**：QEMU启动后进入一个交互式提示符，能敲 `ls`、`cat somefile`、`echo hello` 并看到正确输出。
- **可能踩坑**：
  - 键盘驱动的 scancode→ASCII 映射遗漏特殊键处理。
  - shell 依赖的 read/write syscall 在多任务环境下的阻塞行为没处理好（敲键盘时CPU空转 vs 真正睡眠等待）。
- **学习资料方向**：这一步更多是工程整合，不需要新理论，参考 xv6 的 shell 实现思路即可。

### 阶段 7：驱动与扩展（长期迭代阶段）
- **目标**：从"能跑"到"好用"，按需扩展。
- **模块候选**（按优先级，不必都做）：真实磁盘驱动（ATA/AHCI）+ FAT32、PS/2键盘完善、简单网络栈（可选，难度很高，建议放最后）、简单图形模式（VBE/VGA图形）。
- **完成标准**：视具体扩展目标而定。
- **风险提示**：网络栈和真正的图形界面属于"另一个项目量级"的工作量，务必把它们当作**可选的远期目标**，不要在主线规划里给自己设死限。

## 4. 代码仓库结构设计

```
kumo-os/
├── boot/              # 引导相关：Multiboot2头、GRUB配置(grub.cfg)
├── arch/              # 架构相关代码，按CPU架构分子目录，便于未来移植
│   └── x86/           #   GDT/IDT/分页/上下文切换/usermode切换等汇编+C
├── kernel/            # 内核核心：main.c、syscall分发、内核初始化流程
├── mm/                # 内存管理：物理页帧分配器、分页、内核堆
├── sched/             # 进程/线程调度：TCB、调度算法、上下文切换胶水代码
├── fs/                # 文件系统：VFS抽象层 + 具体实现(ramfs/fat32等)
├── drivers/           # 设备驱动：vga、serial、keyboard、disk等，每设备一子目录
├── user/              # 用户态程序源码（init、简单工具程序）
├── shell/             # shell本身的源码
├── libs/              # 内核态/用户态共用的最小运行时库(kstring、极简libc)
├── tools/             # 构建脚本、镜像打包脚本、交叉工具链构建脚本
├── tests/             # 自动化测试：QEMU+串口输出断言脚本
├── docs/              # 设计文档、阶段笔记、踩坑记录
├── Makefile           # 顶层构建入口
└── linker.ld          # 内核链接脚本
```

**职责说明要点**：
- `arch/` 独立出来是为长期可维护性铺路——即便你现在只做 x86_64，把架构相关代码物理隔离也能强迫自己保持内核核心逻辑与硬件细节解耦，这对调试和未来理解代码都有好处。
- `libs/` 要严格区分"内核态可用的最小运行时"和"用户态libc"，两者不能混用（内核态不能调用为用户态设计的库函数，反之亦然）。
- `docs/` 建议从第一天就写"踩坑记录"，这是个人长期项目最容易丢失的资产——你现在解决的坑，三个月后大概率会忘记怎么解决的。
- `tests/` 不要等到后期才加；从阶段1开始就写"启动后串口应输出XXX"这种自动化断言脚本，靠人眼看QEMU窗口效率很低。

## 5. 第一阶段详细执行方案（阶段0+阶段1，立刻可执行）

### 第一天（环境搭建 Day 1）
1. 安装基础依赖：
   ```bash
   sudo apt update
   sudo apt install build-essential nasm xorriso grub-pc-bin grub-common \
       qemu-system-x86 gdb-multiarch git bison flex libgmp-dev libmpc-dev \
       libmpfr-dev texinfo
   ```
2. 构建交叉编译工具链 `x86_64-elf-gcc`（或先做 `i686-elf-gcc` 配合32位起步方案）：跟着 OSDev Wiki「GCC Cross-Compiler」逐步执行（下载binutils+gcc源码、`./configure --target=i686-elf ...`、编译安装到独立目录、加入PATH）。
   - **产出文件**：`~/opt/cross/bin/i686-elf-gcc` 等一整套交叉工具。
   - **验证方法**：`i686-elf-gcc --version` 正常输出，且 `i686-elf-gcc -print-multi-os-directory` 不报错。
3. 建立仓库骨架：按第4节目录结构 `mkdir -p` 出所有目录，`git init`，写好 `.gitignore`（排除build产物）。

### 第一周
- **Day 2-3**：写最简单的 `boot.asm`（Multiboot2头 + 跳转到 `kmain`）+ `linker.ld` + 空的 `kernel/main.c`（死循环）。用 `grub-mkrescue` 打包成 `.iso`。
  - **产出文件**：`boot/multiboot_header.asm`、`linker.ld`、`kernel/main.c`、`tools/build_iso.sh`
  - **验证**：`qemu-system-i386 -cdrom build/kumo.iso` 不报"no bootable device"，能看到GRUB菜单或直接进入黑屏（说明内核被加载，只是还没输出任何东西——这已经是阶段性胜利）。
- **Day 4-5**：加串口输出驱动（`drivers/serial.c`，操作 COM1 端口 0x3F8），实现 `serial_write_string`。内核启动后打印 "Kumo OS booted."。
  - **产出文件**：`drivers/serial.c`、`drivers/serial.h`
  - **验证**：`qemu-system-i386 -cdrom build/kumo.iso -serial stdio` 后终端里看到打印文本。
- **Day 6**：补 VGA文本模式输出（`drivers/vga.c`）作为视觉验证的补充手段（不作为主调试通道）。
  - **验证**：QEMU图形窗口里能看到彩色文字。
- **Day 7**：整理 `Makefile`，把"改代码→build→跑QEMU"收敛成一条命令（如 `make run`）；写第一篇 `docs/phase1-notes.md` 记录踩坑。
  - **验证**：任何人（包括三个月后的你自己）clone仓库后执行 `make run` 能立刻看到启动打印。

**里程碑判定标准（阶段1完成的硬指标）**：`make run` 一条命令内，从源码到QEMU启动，屏幕/串口能看到内核自己打印的信息，且这个过程可重复、可在干净环境复现。

## 6. 核心模块实现顺序

按以下顺序实现，**不建议打乱**，因为每一步都是后一步的地基：

1. **启动入口（boot.asm + linker.ld）** —— 没有这个什么都跑不起来。
2. **串口/屏幕输出** —— 必须先有输出手段，否则后续所有调试都是盲调。这是"性价比最高"的第一个驱动。
3. **GDT** —— 32/64位保护模式运行的前提，Multiboot启动后GRUB已经设了一个临时GDT，但你需要自己的。
4. **IDT + 中断服务例程 + PIC重映射** —— 有了这个才能安全地捕获异常（否则任何bug都是直接重启，调试效率极低）。
5. **物理内存管理（页帧分配器）** —— 分页的前提是要知道哪些物理页可用。
6. **分页（虚拟内存）** —— 为后续"每个任务独立地址空间"打基础。
7. **内核堆（kmalloc/kfree）** —— 后续几乎所有动态数据结构（任务列表、文件描述符表）都依赖它。
8. **任务切换（context switch）+ 调度器** —— 在内存管理就绪后才能安全地给每个任务分配独立栈/页表。
9. **系统调用 + Ring3切换** —— 需要调度器先存在（要知道"是谁在调用"），也需要分页（用户态地址空间隔离）。
10. **文件系统（先ramfs）** —— 依赖内核堆做缓存，依赖syscall暴露给用户态。
11. **shell** —— 最后一步，是对前面所有子系统的整合调用。

**为什么这个顺序最稳妥**：这本质上是"没有调试手段就没法调试后续任何东西"和"后面模块的数据结构依赖前面模块提供的服务"两条约束的自然结果。历史上几乎所有教学OS项目（xv6、JamesM教程、OSDev Bare Bones系列）都遵循这个大方向，不是巧合。

## 7. 风险评估（诚实版，不画饼）

| 风险 | 说明 | 缓解措施 |
|---|---|---|
| **项目烂尾/半途而废** | 这是个人OS项目最大的风险，没有之一。90%以上的个人OS项目死于"阶段2/3之后热情耗尽"，因为这两个阶段调试反馈最痛苦（三重fault、黑屏无信息）。 | 把每个阶段拆得比本文档还细，每完成一个可见里程碑就写下来（哪怕只是"今天让串口打印出了一个字符"），用可见进度对抗挫败感；设定"每周至少一次可运行的构建"这种硬约束。 |
| **调试黑洞** | x86裸机开发出错时经常没有任何错误信息，只有重启或黑屏，新手极易在这里卡死数天甚至放弃。 | 从阶段1就把QEMU+GDB远程调试(`-s -S`)跑通，并且优先做串口输出（比VGA更适合脚本化验证）；善用QEMU的 `-d int,cpu_reset` 等调试选项看CPU层面发生了什么。 |
| **用 Claude Code 生成代码但自己不理解** | 让AI写OS内核代码效率很高，但如果你不理解GDT/分页/中断这些机制本身，出了问题你根本不知道该怎么问、怎么改，最终会卡在"代码能跑但我不知道为什么"的状态。 | 把AI定位成"结对写代码+讲解"，而不是"自动生成器"；每写完一个模块，尝试自己口头给别人（或对着AI）复述一遍原理，讲不清楚说明还没吃透。 |
| **架构选型中途反悔** | 比如做到阶段3才发现32位/64位选错了，或者高半内核映射方式想推倒重来。 | 阶段0就把架构决策白纸黑字写进 `docs/decisions.md`，做决策评审而不是走到哪算哪；允许阶段1-2内小范围调整，阶段3之后尽量不做架构级返工。 |
| **贪多求全（想同时做网络栈/图形界面/多核）** | 个人项目最常见的自杀式行为：还没有稳定的单核单任务内核就想加SMP、图形界面、TCP/IP。 | 严格按本文档阶段推进，任何"这个功能加进来很酷"的冲动先记进 `docs/backlog.md`，阶段7之后再评估要不要做。 |
| **磁盘文件系统复杂度被低估** | 真实文件系统（FAT32等）+ 真实磁盘驱动（ATA/AHCI）比大多数人预期的复杂得多，尤其是DMA、中断驱动的磁盘I/O。 | 阶段5坚决先用ramfs，把"有文件系统接口能跑通"和"数据真正持久化到磁盘"分成两个明确不同的里程碑，不要合并。 |
| **测试覆盖不足导致回归** | 内核代码改一个模块经常意外破坏另一个模块（比如改调度器把中断处理弄坏了），没有自动化测试会导致"改了却不知道改坏了什么"。 | 从阶段1起维护一套QEMU自动化冒烟测试（串口输出关键字断言），每次改动前后跑一遍。 |

**最诚实的一句话总结**：这个项目**技术上完全可行**（大量个人开发者和教学项目已验证路径），但**能不能做完主要取决于你能不能扛过阶段2-3那段"每天调试到深夜，屏幕却什么都不显示"的低反馈期**。如果你了解自己容易在这种阶段放弃，建议提前给自己设计"小胜利"节奏（比如每周五给自己一个明确的、哪怕很小的可见输出目标）。

## 8. 学习路线（按优先级排序）

假设你现在基础一般（但根据你过去的技术背景——HarmonyOS ArkTS开发、ML训练、游戏开发，你的工程能力和学习能力应该是足够的，欠缺的更多是底层系统知识而非编程能力本身）：

1. **C语言底层**（最优先，且是硬门槛）
   - 重点：指针运算、结构体内存布局、位运算、`volatile`关键字、函数指针、内联汇编基础语法。
   - 不需要：C++那套东西完全不需要，甚至要刻意避免用C++思维写C。
2. **计算机组成原理**（第二优先）
   - 重点：CPU寄存器、栈帧、内存地址空间概念、中断的硬件原理、I/O端口 vs 内存映射I/O。
   - 这是理解"为什么要有GDT/IDT"的地基，跳过这块直接抄代码会导致完全不理解自己在做什么。
3. **x86(_64) 架构**（跟第2项紧密配合，交替学）
   - 重点：实模式→保护模式→长模式的演进逻辑、段寻址与分页寻址、特权级(Ring)概念、GDT/LDT/TSS的作用。
   - 资料：Intel SDM（不用通读，按需查阅Vol.3）、OSDev Wiki作为主线教程。
4. **汇编基础**（不需要精通，够用即可）
   - 重点：只需要能读懂并写出简单的NASM代码（跳转、栈操作、中断相关指令如`cli/sti/iret`），不需要写复杂算法。
5. **操作系统原理**（与实践同步进行，不要脱离实践单独啃理论）
   - 重点：进程/线程概念、调度算法基础（先懂轮转即可，不用一开始学CFS这种复杂算法）、虚拟内存与分页、文件系统基本概念（inode/目录树）。
   - 资料：《操作系统导论》(OSTEP，有免费在线版) 是目前公认最适合自学者的教材，配合xv6源码阅读效果最好。
6. **链接与加载**
   - 重点：ELF文件格式基础、链接脚本(linker script)语法、为什么内核需要自定义链接脚本。
   - 这块可以放最后，等阶段1实践中遇到问题再针对性查。
7. **裸机开发（freestanding）经验**
   - 这不是一门独立的"知识"，而是前面所有知识在"没有操作系统支撑"这个约束下的综合应用，通过阶段0-2的实践自然获得，不需要单独系统学习。

**学习方式建议**：不要试图"学完再动手"，这几乎必然导致半途而废。正确节奏是"遇到阶段1需要理解Multiboot头→现学Multiboot规范和ELF基础→写代码验证→进阶段2再学中断"，理论追着实践走，而不是反过来。
