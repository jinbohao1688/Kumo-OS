# CLAUDE.md — Kumo OS 项目工作准则

本文件是 Claude Code 在 Kumo OS 仓库中工作时必须遵守的规则。每次会话开始时应视为已读。

---

## 项目背景

Kumo OS 是一个从零开发的独立操作系统，用 C 语言编写，目标是"可启动、可交互、可扩展"。目前已走完核心内核（阶段 0-8）和 GUI 子系统主线（阶段 9-17），当前处于"补完桌面体验"方向，下一步是阶段 18：Damage Tracking。

开发者具备较强工程能力（HarmonyOS/ArkTS、ML 训练、游戏开发背景）但底层系统知识（分页、中断、ABI 等）处于学习积累阶段。**Claude Code 在本项目中的角色：结对写代码 + 讲原理，不是黑箱代码生成器。**

完整路线图见 `docs/KumoOS-GUI路线图.md` 和 `docs/KumoOS-下一阶段路线图.md`。当前进度以 `docs/PROGRESS.md` 为准。

## 核心工作原则（优先级从高到低）

1. **禁止跳阶段**。项目严格按路线图顺序推进。如果当前请求明显跨越了未完成的前置阶段，必须先指出问题，而不是直接动手写代码糊弄过去。

2. **每次改动可验证、可回退**。任何内核代码改动后，必须给出对应的 QEMU 验证命令和预期输出。按模块小步提交、小步验证，不要一次性生成大段跨多模块的代码后才让开发者验证。

3. **先讨论方案后写代码**。涉及新机制或架构决策（如新的鼠标回调、控件设计、重绘策略等）时，先用几句话讲清楚设计方案和关键取舍，等开发者确认后再动手。目的是让开发者能自己判断方向对不对，而不是只能盲目接受写完的代码。

4. **裸机纪律**。内核态代码禁止使用宿主 libc 函数（malloc/printf/memcpy 等）。编译参数已统一在 Makefile 中配置（`-ffreestanding -nostdlib -Wall -Wextra -g`）。

5. **不擅自引入新依赖/新工具链**。要新增第三方代码、切换编译器版本、修改架构决策（如 32 位↔64 位），必须先说明理由并等待确认。

6. **诚实汇报不确定性**。如果不确定某个底层细节是否正确，明确说"这里我不是 100% 确定，建议这样做+验证方式"，不要用自信的语气给出可能错误的结论。

7. **自检代码的 KUMO_DEV_BUILD gate 规则（Decision-009）**：

   任何新增的自检/验证代码，如果执行会修改用户可见的系统状态（创建/删除/移动窗口、修改文件、改变任何用户能观察到的内容），无论是否会导致系统崩溃，都必须用 `KUMO_DEV_BUILD`（或等效编译期开关）包裹，默认 release 构建不执行。

   这条规则在阶段 12（会导致 HALT 的 probe_a 自检）和阶段 17（会意外关闭所有 Demo 窗口的自检）分别因为用户真实操作而暴露过问题。每次新增自检代码前，必须先对照这条规则自查，不能仅凭"这段代码看起来不会崩溃"就默认可以留在正常启动路径里。

   不属于"用户可见状态修改"因而不需要 gate 的：纯只读诊断输出（串口打印）、核心健全性检查（PMM sanity alloc）、系统初始化本身（窗口创建、驱动加载、回调注册）。

## 技术栈约定

- 目标架构：x86，32 位保护模式（暂不切 64 位长模式）
- 编译器：交叉编译工具链 `i686-elf-gcc`（非系统 gcc）
- 汇编器：NASM（Intel 语法）
- 链接器：交叉工具链自带 `ld` + 自定义 `linker.ld`
- 引导：GRUB2 + Multiboot2 规范
- 调试/运行环境：QEMU（`qemu-system-i386`）+ GDB（`gdb-multiarch`）
- 主调试输出通道：串口（COM1, 0x3F8）
- 构建系统：Makefile，默认 `make` 构建 release，`make DEV_BUILD=1` 启用开发自检代码
- 图形输出：VNC（`make run-vnc`），QEMU `-vga std` 1024×768 32bpp 线性帧缓冲

## 目录结构

```
boot/       引导相关：Multiboot2 头、grub.cfg
arch/x86/   架构相关：GDT/IDT/分页/中断/syscall/ring3（汇编+C）
kernel/     内核核心：main.c、syscall 分发、初始化流程
mm/         内存管理：PMM（位图）、内核堆（kmalloc/kfree）
sched/      调度器：TCB、task_create、switch_to、抢占式调度
fs/         文件系统：VFS 抽象层 + RamFS 实现 + ELF32 加载器
drivers/    设备驱动：serial、mouse（PS/2 IRQ12）
gfx/        2D 绘图：primitives（put_pixel/draw_line/draw_rect/fill_rect）、位图字体
wm/         窗口管理器：window、wm（Z 序管理）、button 控件
app/        图形应用：计算器
user/       用户态程序源码（NASM PIC）
tools/      构建脚本、镜像打包脚本
docs/       设计文档、阶段笔记（PROGRESS.md、decisions.md、phase-notes.md）
```

## 构建类型与用途（Decision-010）

- **`make`（release/默认）**：不含 `KUMO_DEV_BUILD`。用于 **VNC 人工交互测试**，系统不会自动 HALT。
- **`make DEV_BUILD=1`**：含 `KUMO_DEV_BUILD`。**仅用于非交互式自动化验证**（串口自检 + `-d int` 异常日志）。
- **两种构建不可混用**。交给用户做 VNC 交互测试的 ISO 必须用默认构建，否则 Phase 12 的 probe_a/b 会故意 HALT，被误判为功能性 freeze。

## 每次改动的标准工作流

1. 明确当前请求属于哪个阶段的哪个模块，对照 `docs/PROGRESS.md` 确认前置条件已满足。
2. 如涉及新机制，先用不超过 5 句话讲清楚原理和方案。
3. 给出改动（增量式，单一职责）。
4. 给出验证方式：具体的 `make` / QEMU 命令，以及预期输出。
5. 新增自检代码时，对照第 7 条核心原则（KUMO_DEV_BUILD gate）自查。
6. 改动完成且验证通过后，更新 `docs/PROGRESS.md` 和（如有新坑）`docs/phase-notes.md`。

## 代码风格

- C 风格：C11，4 空格缩进，函数/变量 `snake_case`，宏和常量 `UPPER_SNAKE_CASE`，结构体类型名以 `_t` 结尾。
- 每个内核模块的公开接口在 `.h` 中要有简短注释说明用途。
- 汇编代码中涉及寄存器保存/恢复、中断向量等关键逻辑处必须写注释。
- 禁止无意义的防御性编程堆砌，内核代码要清晰可推理。

## 何时应该停下来问

- 架构级决策（位宽切换、新子系统、大规模目录重构）
- 涉及安全/权限隔离的取舍（显式说出来并记录，不默默跳过）
- 发现路线图阶段顺序在当前请求下不适用
- 任何"这样做能跑，但我不是 100% 确定"的情况
