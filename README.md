# Kumo OS

一个用 C 语言 + NASM 汇编从零开发的 x86 32位教学级操作系统内核，从引导（bootloader）到图形界面（GUI）的完整技术栈。

这是一个与 [Claude Code](https://claude.com/claude-code) 协作完成的个人学习项目。全部代码经过实机（QEMU/VMware）验证，开发过程中记录了完整的架构决策和踩坑经验（见 `docs/` 目录）。

---

## 这个系统能做什么

启动后会看到一个图形化桌面：三个可拖动、可点击置顶的窗口，加上一个能做真实四则运算的图形计算器（点击按钮输入 `5 + 3 =`，正确显示 `8`）。窗口可以用鼠标拖动标题栏来移动位置。

背后是一套从裸机一路搭起来的完整内核：

```
┌─────────────────────────────────────────┐
│  用户态：shell、ELF 程序                  │  Ring 3
├─────────────────────────────────────────┤
│  系统调用接口 (int 0x80)                  │
├─────────────────────────────────────────┤
│  内核子系统：                              │  Ring 0
│  ├─ 进程调度（协作式 + 抢占式）             │
│  ├─ 每任务独立页目录（进程隔离）            │
│  ├─ VFS + RamFS 文件系统                  │
│  ├─ 窗口系统 / 2D 图形 / 按钮控件          │
│  └─ 中断/异常处理、8259 PIC               │
├─────────────────────────────────────────┤
│  内存管理：物理页帧分配器 + 分页 + 内核堆    │
├─────────────────────────────────────────┤
│  引导：GRUB2 (Multiboot2) → 内核入口       │
└─────────────────────────────────────────┘
```

## 功能特性

### 核心内核（阶段 0–8）
- GRUB2 (Multiboot2) 引导，32位保护模式
- GDT / IDT / 8259 PIC 重映射 / 完整的中断与异常处理（32 个 CPU 异常向量全部实测验证）
- 物理内存管理（位图分配器）+ 分页（恒等映射）+ 内核堆（链表式分配器，支持前向/后向合并）
- 多任务调度（协作式 → 抢占式）
- Ring 0 / Ring 3 特权级切换 + 系统调用（int 0x80）
- 用户态内存安全校验（`copy_from_user` / `copy_to_user`，防止内核态被恶意/错误指针破坏）
- VFS + RamFS 文件系统
- Shell（内建命令：help / ls / cat / echo / run / exec）
- ELF32 可执行文件加载器

### GUI 扩展（阶段 9–16）
- Framebuffer 图形驱动（Multiboot2 framebuffer tag）
- 2D 绘图原语（像素/直线/矩形）+ 自制 8×16 位图字体（无第三方版权依赖）
- PS/2 鼠标驱动（IRQ12，完整寄存器现场保护）
- 每任务独立页目录 —— 真正的进程内存隔离
- 窗口系统：多窗口 Z 序合成、点击置顶、窗口拖动
- 按钮控件（符合行业标准交互手势：释放位置决定触发/取消，与 Windows/macOS/GTK/Qt 行为一致）
- 图形计算器应用（内核态实现，验证完整的窗口 + 控件 + 应用分层架构）

## 工程亮点

这个项目的每一层机制都有**实测验证证据**支撑，不是"写完看起来能跑就算完成"：

- **进程隔离**：两个用户态程序互相探测对方内存，跨任务访问精确触发 `#PF`，`ErrCode=0x05`（保护违规而非缺页）证明按需克隆页表机制真正生效，不是巧合触发的异常。
- **中断异常处理**：32 个 CPU 异常向量（含 error code 的有无）逐一核对 Intel SDM 规范，用 `#DE` 和 `#GP` 分别实测验证了两种不同栈布局路径的正确性。
- **发现并修复过的严重问题**包括：`switch_to` 上下文切换栈布局方向性错误、页表 PDE/PTE 权限位遗漏导致的用户态越权访问、`draw_line` 在负坐标输入下的无限循环风险（发生在中断上下文中，一旦触发会导致系统彻底挂死且无法自动恢复）。

## 如何构建运行

### 环境要求
- Ubuntu 22.04（或类似发行版）
- 交叉编译工具链：`i686-elf-gcc` + `i686-elf-ld`（32位保护模式目标）
- NASM、GRUB2 工具（`grub-mkrescue`）、`xorriso`
- QEMU（`qemu-system-i386`）

### 构建 + 运行
```bash
make run          # 编译 + 生成 ISO + 启动 QEMU（串口输出直接在终端显示）
```

### 图形界面（VNC）
```bash
make run-vnc       # 启动 QEMU，图形输出走 VNC (端口 5900)，串口日志仍在终端显示
```
用任意 VNC 客户端（如 Remmina、TigerVNC）连接 `localhost:5900` 即可看到图形桌面并用鼠标操作。

### 直接运行发行版镜像
Releases 页面提供预构建的 `.iso` 镜像，可以直接用 QEMU 或 VirtualBox / VMware 等虚拟机软件挂载启动。

```
boot/       引导相关：Multiboot2 头、GRUB 配置
arch/x86/   架构相关：GDT/IDT/分页/上下文切换/中断服务例程
kernel/     内核核心：初始化流程、syscall 分发
mm/         内存管理：物理页帧分配器、内核堆
drivers/    设备驱动：串口、VGA、framebuffer、PS/2 鼠标
gfx/        2D 图形原语、位图字体
wm/         窗口管理器：窗口、Z序合成、按钮控件
app/        图形应用：计算器
fs/         文件系统：VFS + RamFS、ELF 加载器
sched/      调度器：任务控制块、上下文切换
shell/      Shell 实现
libs/       内核态最小运行时库
docs/       架构决策记录（decisions.md）、踩坑经验（phase-notes.md）、
            里程碑回顾、GUI 路线图
```

## 开发过程

这个项目通过与 Claude Code 结对开发完成，采用"先讨论方案、后写代码、每一步都要求真实验证证据"的工作方式——不满足于"编译通过、看起来能跑"，而是要求 GDB 单步核对寄存器/内存内容、error code 精确匹配、像素级采样验证等具体证据。多次在方案讨论阶段就发现并修正了潜在的严重 bug（例如上下文切换栈布局方向错误、跨页边界的指针校验逻辑漏洞、release 镜像默认包含会导致系统挂起的开发期自检代码等），避免了它们被写进代码后才在更晚的阶段暴露。

完整的开发过程记录在 `docs/` 目录：
- `decisions.md` —— 逐条编号的架构决策记录（ADR），包括每个决策的理由和已知的技术债
- `phase-notes.md` —— 踩坑经验记录，包含具体的问题现象、根因和可迁移的经验教训
- `milestone-review-*.md` —— 每个大阶段完成后的里程碑技术回顾

感兴趣的话，可以从这些文档了解具体的工程决策过程，而不只是看最终代码。

## Roadmap

GUI 相关的骨架能力（窗口、图形、鼠标交互）已经完成到"简单 hobby OS"级别（对标 SerenityOS / ToaruOS 早期阶段的目标）。后续方向包括：

- 补完桌面体验：窗口关闭按钮、Damage Tracking（脏矩形重绘，优化当前的全屏重绘性能问题）
- 键盘驱动 + 文本输入控件
- 持久化存储（真实磁盘驱动 + 文件系统，当前的 RamFS 关机即清空）
- 用户态 GUI 架构迁移（当前所有窗口/绘图逻辑运行在内核态，是权宜之计）

这是一个持续开发的个人项目，详细的方向规划见 `docs/KumoOS-下一阶段路线图.md`。
