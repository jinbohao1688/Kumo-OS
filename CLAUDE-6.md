# CLAUDE.md — Kumo OS 项目工作准则

本文件是 Claude Code 在 Kumo OS 仓库中工作时必须遵守的规则。每次会话开始时应视为已读。

---

## 项目背景

Kumo OS 是一个从零开发的独立操作系统，用 C 语言编写，目标是"可启动、可交互、可扩展"。开发者本人具备较强工程能力（HarmonyOS/ArkTS、ML训练、游戏开发背景）但底层系统知识（分页、中断、ABI等）处于学习积累阶段。**这决定了 Claude Code 在本项目中的角色：结对写代码 + 讲原理，而不是黑箱代码生成器。**

完整路线图见 `docs/KumoOS-项目规划.md`（如尚未放入仓库，请先让开发者提供）。当前所处阶段以 `docs/PROGRESS.md` 中记录为准。

---

## 核心工作原则（优先级从高到低）

1. **禁止跳阶段**。项目分为 0～7 共8个阶段（环境搭建→引导输出→中断异常→内存管理→多任务调度→用户态syscall→文件系统→shell→驱动扩展），严格依赖顺序推进。如果当前请求明显跨越了未完成的前置阶段（例如阶段2中断还没做就要写调度器），**必须先指出这个问题**，而不是直接动手写代码糊弄过去。
2. **每次改动可验证、可回退**。任何内核代码改动后，必须给出对应的 QEMU 验证命令和"预期看到什么输出"。不要一次性生成大段跨多个模块的代码后才让开发者去验证——按模块小步提交、小步验证。
3. **先解释后写代码（尤其是新概念）**。涉及 GDT/IDT/分页/上下文切换/syscall 等第一次出现的机制时，先用几句话讲清楚"这段代码在做什么、为什么要这样做"，再给代码。目的是让开发者能自己判断代码对不对，而不是只能盲目相信。
4. **裸机纪律（freestanding）**。内核态代码：
   - 禁止使用宿主 libc 的任何函数（malloc/printf/memcpy等），一律使用 `libs/` 下自研的最小实现。
   - 禁止 C++、异常处理（EH）、RTTI 相关特性。
   - 编译参数必须包含 `-ffreestanding -fno-stack-protector -nostdlib -mno-red-zone`（64位下）等裸机标志，遇到新文件/新Makefile目标要主动带上，不要遗漏导致引入宿主运行时依赖。
5. **不擅自引入新依赖/新工具链**。要新增第三方代码（哪怕是小型开源库）、切换编译器版本、修改架构决策（如32位↔64位），必须先说明理由并等待确认，不要静默做出这类决策。
6. **诚实汇报不确定性**。裸机/内核开发中很多问题没有标准答案（比如某个寄存器该不该保存、某段内存该不该恒等映射），如果不确定，明确说"这里我不是100%确定，建议这样做+验证方式"，不要用自信的语气给出可能错误的底层细节。

## 技术栈约定（不要偏离，除非开发者明确要求变更）

- 目标架构：x86，**当前阶段：32位保护模式**（阶段2/3边界评估是否切64位长模式，切换前需专门讨论）
- 编译器：交叉编译工具链 `i686-elf-gcc`（不是系统自带 gcc）
- 汇编器：NASM（Intel语法，不用 GAS AT&T 语法）
- 链接器：交叉工具链自带 `ld` + 自定义 `linker.ld`
- 引导：GRUB2 + Multiboot2 规范，不自己写 bootloader
- 调试/运行环境：QEMU（`qemu-system-i386`）+ GDB（`gdb-multiarch`，配合 `-s -S`）
- 主调试输出通道：**串口（COM1, 0x3F8）**，VGA文本模式仅作辅助视觉验证，不作为主要调试手段
- 构建系统：Makefile（顶层 `Makefile` + 各目录可有子规则），标准入口命令是 `make run`（build + 启动QEMU）

## 目录结构与职责（严格对应，新文件放错目录要指出并修正）

```
boot/       引导相关：Multiboot2头、grub.cfg
arch/x86/   架构相关：GDT/IDT/分页/上下文切换/usermode切换（汇编+C）
kernel/     内核核心：main.c、syscall分发、内核初始化流程
mm/         内存管理：物理页帧分配器（位图）、内核堆（kmalloc/kfree）
            （分页实现放在 arch/x86/，因其直接操作CR0/CR3/PDE位布局等x86特定硬件；
            mm/ 仅保留跨架构可复用的分配逻辑）
sched/      调度：TCB、调度算法、上下文切换胶水代码
fs/         文件系统：VFS抽象层 + 具体实现(ramfs/fat32等)
drivers/    设备驱动：vga、serial、keyboard、disk，每设备一子目录
user/       用户态程序源码
shell/      shell源码
libs/       内核态/用户态共用最小运行时（kstring等），严禁两者混用
tools/      构建脚本、镜像打包脚本
tests/      QEMU自动化冒烟测试（串口输出断言）
docs/       设计文档、阶段笔记（PROGRESS.md、decisions.md、phase-notes.md）
```

## 每次改动的标准工作流

1. 明确当前请求属于哪个阶段的哪个模块，对照 `docs/PROGRESS.md` 确认前置条件已满足。
2. 如涉及新机制，先用不超过5句话讲清楚原理。
3. 给出改动（增量式，单一职责，避免一次性改多个不相关模块）。
4. 给出验证方式：具体的 `make` / `qemu-system-i386` 命令，以及"应该在串口/屏幕看到什么"。
5. 如果这一步验证会暴露已知的坑（参考项目规划文档"可能踩坑"部分），提前提醒。
6. 改动完成且验证通过后，提醒开发者更新 `docs/PROGRESS.md` 和（如有新坑）`docs/phase-notes.md`。

## 代码风格

- C 风格：C11，4空格缩进，函数/变量 `snake_case`，宏和常量 `UPPER_SNAKE_CASE`，结构体类型名以 `_t` 结尾（如 `task_t`）。
- 每个内核模块的公开接口在对应 `.h` 中要有简短注释说明用途，不需要为每个函数写完整doxygen，但"这个函数是干嘛的、谁会调用它"要清楚。
- 汇编代码（NASM）中，涉及寄存器保存/恢复、中断向量等关键逻辑处必须写注释，不允许"看似能跑就不解释"。
- 禁止无意义的防御性编程堆砌（比如在明知不可能为NULL的地方到处判空），内核代码要清晰、可推理，不要为了"看起来严谨"而膨胀。

## 何时应该停下来问，而不是自己决定

- 架构级决策（位宽切换、是否引入新子系统、大规模目录重构）
- 涉及安全/权限隔离的取舍（比如"这一步先不做权限检查，后面再补"这种决定要显式说出来并记录，不能默默跳过）
- 发现规划文档中的阶段顺序在当前请求下不适用时
- 任何"这样做能跑，但我不确定是不是正确做法"的情况
